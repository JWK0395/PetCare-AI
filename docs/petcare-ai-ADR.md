## PET-001 · 반려동물 프로필 등록

| 항목 | 내용 |
| --- | --- |
| 사용자 스토리 | 보호자로서 반려동물의 기본 정보와 건강 배경을 등록하고 싶다. |
| 입력 | 이름, 종, 품종, 성별, 중성화, 생년월일, 체중, 알레르기, 질환, 복용약 |
| 처리 | 입력 검증 → 저장 → 초기 컨텍스트 생성 |
| 출력 | 프로필 요약 |
| 예외 | 미래 생년월일, 음수 체중, 지원하지 않는 종 |
| 상태 | 작성중 → 검증 → 저장 |
| 이벤트 | `pet_profile_completed` |

### 인수 기준

- 필수값이 유효하면 프로필이 생성된다.
- 알레르기와 복용약은 상태 체크에 사용된다.
- 다른 사용자의 반려동물 데이터에는 접근할 수 없다.

## HOME-001 · 홈 대시보드

| 항목 | 내용 |
| --- | --- |
| 사용자 스토리 | 오늘 기록 여부와 최근 변화를 한눈에 보고 싶다. |
| 입력 | 선택된 반려동물 |
| 처리 | 프로필, 오늘 기록, 최근 알림, 일정 조회 |
| 출력 | 프로필 카드, 기록 버튼, 최근 변화, 예정 일정 |
| 예외 | 기록 없음, 기준선 부족 |
| 이벤트 | `home_viewed` |

### 인수 기준

- 오늘 기록이 없으면 빠른 기록 버튼을 표시한다.
- 기준선이 부족하면 정상 범위를 표시하지 않는다.
- 주의 변화에는 기간과 변화 방향을 표시한다.

## LOG-001 · 자연어 건강일기 구조화

| 항목 | 내용 |
| --- | --- |
| 사용자 스토리 | 일상 문장을 건강기록으로 자동 정리하고 싶다. |
| 입력 | 자연어 일기, 기록 날짜 |
| 처리 | 증상, 수치, 활동, 시간 추출 |
| 출력 | 구조화 후보, 신뢰도, 원문 근거 |
| 예외 | 모호한 수치, 충돌 표현, 분석 실패 |
| 상태 | 입력 → 분석중 → 검토 |
| 이벤트 | `diary_extracted` |

### 인수 기준

- “노란 토를 한 번 했다”는 구토 1회, 노란색으로 추출한다.
- “사료를 반쯤 남겼다”는 근거 없이 g 단위로 변환하지 않는다.
- 모든 후보에 원문 근거를 표시한다.

## CHECK-001 · AI 상태 체크

| 항목 | 내용 |
| --- | --- |
| 사용자 스토리 | 증상을 입력하면 최근 기록을 반영한 행동 단계를 받고 싶다. |
| 입력 | 자연어 증상 |
| 처리 | 컨텍스트 조회 → 응급 검사 → 추세 분석 → 추가 질문 → RAG |
| 출력 | 위험 단계, 근거, 추가 질문, 행동 안내 |
| 예외 | 기록 부족, 근거 없음, LLM 오류 |
| 상태 | 시작 → 질문 → 분석 → 완료 |
| 이벤트 | `health_check_completed` |

### 인수 기준

- 응급 규칙에 일치하면 일반 경로를 중단한다.
- 최근 변화에는 기준선 기간과 변화율을 표시한다.
- 정보가 부족하면 정상 판정을 하지 않는다.
- 외부 의료 설명에는 인용을 표시한다.

## EMER-001 · 응급 경로

| 항목 | 내용 |
| --- | --- |
| 사용자 스토리 | 생명 위협 가능성이 있으면 즉시 병원 연락 안내를 받고 싶다. |
| 입력 | 자연어 증상, 구조화 증상 |
| 처리 | 응급 규칙 검사 → 즉시 경고 → 병원 및 문서 CTA |
| 출력 | 응급 경고, 전화 버튼, 병원 후보, 이동 중 안내 |
| 예외 | 병원 정보 오래됨, 위치 권한 없음 |
| 상태 | 응급 감지 → 경고 → 연락 |
| 이벤트 | `emergency_detected` |

### 인수 기준

- 호흡곤란과 청색증 표현은 일반 질문보다 먼저 응급 경고를 표시한다.
- 즉시 병원에 연락하라는 행동을 제공한다.
- 검증되지 않은 병원 정보는 현재 진료 중으로 확정하지 않는다.
- 사용된 응급 규칙 버전을 내부 기록에 남긴다.

## DOC-001 · 진단서 PDF 등록

| 항목 | 내용 |
| --- | --- |
| 사용자 스토리 | 진단서 PDF의 주요 정보를 건강기록으로 남기고 싶다. |
| 입력 | PDF |
| 처리 | 파일 검사 → 저장 → OCR/텍스트 추출 → 필드 추출 → 사용자 확인 |
| 출력 | 병원, 수의사, 날짜, 진단, 처방 후보 |
| 예외 | 암호 PDF, 빈 파일, 저품질 스캔, 악성 파일 |
| 상태 | 업로드 → 추출 → 검토 → 저장 |
| 이벤트 | `document_confirmed` |

### 인수 기준

- 각 필드에 원문 페이지 또는 근거 위치를 표시한다.
- 사용자 승인값만 정식 진료 이벤트에 저장한다.
- 원본 파일 해시와 추출 버전을 기록한다.
- 처방 내용은 시스템 권고가 아니라 문서 원문으로 저장한다.

- 사용자 입력과 시스템 계산을 구분한다.
- 기록 누락을 미확인 항목으로 표시한다.
- 데이터 기간과 생성시각을 표시한다.
- 사용자가 수정한 뒤 PDF를 생성한다.
- 새로운 진단명을 추론하지 않는다.

## 공통 UX 규칙

1. 위험 단계는 색상뿐 아니라 텍스트와 아이콘으로 표시한다.
2. 사용자 입력, 시스템 계산, 외부 근거를 서로 다른 라벨로 표시한다.
3. AI 추출값에는 원문 근거와 수정 기능을 제공한다.
4. 데이터가 없을 때 0이나 정상으로 표현하지 않는다.
5. 응급 화면에서는 전화와 병원 전달 행동을 가장 먼저 배치한다.
6. 병원 정보에는 출처와 갱신시각을 표시한다.
7. 이메일과 문서 삭제 같은 민감 행동에는 명확한 결과 피드백을 제공한다.
8. 모든 의료적 설명은 확인된 정보, 위험 가능성, 권장 행동 순서로 표현한다.
